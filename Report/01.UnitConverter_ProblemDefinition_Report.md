## 1. 배경

소형 가구 제작·해외 DIY 자재 리셀 등 **실무 현장**에서는 수입 카탈로그(feet/yard/inch)와 국내 발주·택배·재단 기준(m/mm)이 **한 건 안에 섞인다**.  
`UnitConverter.py`는 길이 단위 변환 CLI이나, Mom Test 결과 **현재 구현은 진짜 문제의 일부(환산)만** 다루며 **업무 흐름·판정·연속 처리**와는 gap이 있다.

### 현재 구현 SNAPSHOT (`UnitConverter.py`)

| 항목 | 내용 |
|------|------|
| 입력 | `unit:value` (예: `meter:2.5`) |
| 지원 단위 | meter, feet, yard |
| 미지원 | inch, mm, cm 등 |
| 실행 | 1회 1입력 → 3줄 출력 → 종료 |
| 비율 | `1 m = 3.28084 ft`, `1 m = 1.09361 yd` |
| 검증 | 콜론 형식, 숫자 파싱, unknown unit (음수 검증 **미구현**) |

## 2. 사용자·맥락 (Mom Test)

### 2.1 주요 페르소나

| | 박지현 (주 인터뷰) | 이준호 (시뮬레이션) |
|---|-------------------|---------------------|
| **역할** | 소형 가구 공방 실무 | 1인 해외 DIY 자재 리셀 |
| **단위 맥락** | 수입 몰딩 feet/yard/inch, 국내 m/mm | 인보이스 feet/yard, 택배 cm |
| **도구 사용** | converter **4회** (2026.5~6) | converter **3회** (2026.4~5) |

### 2.2 공통 행동 패턴 (과거 사실)

- 카탈로그·메모에 **숫자만** 기록 (**"6"**, **"2.5"**, **"900"**)
- 발주서 **"길이(m)"** 칸에 **단위 확인 없이** 숫자 복사
- 환산 시 **구글 검색 + 엑셀 + 프로그램 재실행** 반복
- **feet / yard / inch / meter**가 **한 건**에 동시 등장 (주문 **22건 중 3건**)

## 3. 문제 정의

### 3.1 표면 문제 (잘못된 정의 — 사용 금지)

솔루션이 이미 섞인 문장. **요구사항·PRD 본문에 그대로 쓰지 않는다.**

- "meter·feet·yard를 **한 번에 변환해 주는 프로그램**을 만든다."
- "**숫자만 입력**해도 변환되는 converter를 만든다."
- "**inch·mm까지 지원**하는 UnitConverter를 만든다."

### 3.2 진짜 문제 (한 문장)

**수입·국내 자재를 견적·발주할 때 feet/yard/inch/m/mm 단위가 카탈로그·발주서·현장 메모마다 섞여, 숫자만 옮기거나 환산을 반복·교차 확인하다 시간을 쓰고, 단위를 끝까지 확인하지 않으면 발주 오류와 재발주·공기 손실로 이어진다.**

### 3.3 Mom Test 증거

1. "카탈로그 **'2.5 yard / roll'** … 발주서 **'길이(m)'** 칸에 **2.5**를 그대로 입력."
2. "**yard를 meter로 잘못 읽어** … **약 35만 원** 손해."
3. "**1시간 20분**" / "`feet:12` … **종료** → `yard:2.5` … **약 7분**"

## 4. R-G-I-O

### 주제 (1문장)

**수입·국내 길이 단위가 한 발주 건 안에서 섞일 때, 견적·발주 전 치수를 하나의 기준으로 맞추고 판정하는 일.**

| | 내용 |
|---|------|
| **R — Reality** | 카탈로그·인보이스(feet/yard/inch) vs 발주·택배·재단(m/mm). 메모는 단위 없이 적는 경우 잦음. |
| **G — Goal** | 견적·발주·발송 **판정 전** 치수를 **m 또는 mm** 기준으로 맞춰 **맞다/틀리다/초과** 판정. |
| **I — Issue** | 출처별 단위 불일치 + 숫자만 옮김 + 검색·엑셀·재실행 반복 + 단위 미확인 시 yard↔m 혼동. |
| **O — Outcome** | 한 건 내 다중 치수 **빠른 대조**, 단위 미확인 발주 **없음**, 재발주·공기 손실 **재발 없음**. |

### Issue ↔ 과거 사실 (기준선)

| 증상 | 사실 |
|------|------|
| 단위 혼동 발주 | **2025-11-07**, **약 35만 원** |
| 다단위 1건 | **2026-05-20**, **1시간 20분** |
| 재실행·형식 gap | **5/28** 7~15분; **6/3** `Invalid format`; **inch** `Unknown unit` |
| 출력 후처리 | **6/5** mm 환산 **5분** |

## 5. 성공 기준

| # | 기준 | 측정 | Mom Test 기준선 |
|---|------|------|-----------------|
| 1 | 단위 혼동 발주 **0건** | 3개월, 숫자만 복사 재발주 0 | 35만 원 (11/7) |
| 2 | 다단위 1건 **≤ 20분** | feet+yard(+inch) 환산·대조 | 1시간 20분 (5/20) |
| 3 | 단위 없는 메모 **판정 가능** | `2.5`, `6` 등 → m/mm (Invalid 0 또는 1단계) | 6/3 실패 |
| 4 | **연속 처리** | 동일 건 2+ 치수, 재시작 없음 | 5/28 2회 재실행 |
| 5 | **업무 출력 일치** | m 소수 2자리 또는 mm 정수, 추가 계산 없음 | 5~3분 후처리 |

## 6. 현재 구현 vs 진짜 문제 (Gap)

| 진짜 문제 / 성공 기준 | 현재 `UnitConverter.py` | Gap |
|----------------------|-------------------------|-----|
| #1 단위 혼동 방지 | 단위 **명시** 입력 필요 | 숫자만 메모 습관과 불일치 |
| #2 다단위 20분 | meter/feet/yard 3단위만 | inch/mm·다품목 미커버 |
| #3 단위 없는 입력 | `:` 없으면 **Invalid** | `2.5` 거부 |
| #4 연속 처리 | **1실행 1입력** | 재실행 필요 |
| #5 m/mm 출력 | raw float 3줄 | 자릿수·mm 없음 |
