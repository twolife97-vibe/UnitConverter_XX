# UnitConverter_XX — STEP 2 R-G-I-O 프롬프트

## 1. 실행 프롬프트

```
진짜 문제 → R-G-I-O + 주제 1문장 + 성공 기준

입력:
- STEP 1 Mom Test 진짜 문제 (한 문장)
- Mom Test 증거 3줄
- UnitConverter.py 현재 구현 제약 (선택)

출력 형식:
1. 주제 (1문장)
2. R-G-I-O 표
   - R — Reality (현실): 누가, 어떤 맥락, 어떤 단위가 섞이는지
   - G — Goal (목표): 사용자가 달성하려는 일 (솔루션名 없음)
   - I — Issue (핵심 장애): 불편·반복·혼동 (과거 증거로 뒷받침)
   - O — Outcome (바라는 결과): 원하는 상태 (제품 기능 나열 금지)
3. 성공 기준 (표)
   - Mom Test 과거 수치를 기준선으로
   - 측정 가능 (시간, 건수, 오류 0건 등)
   - 3~5개

규칙:
- 진짜 문제·R-G-I-O·성공 기준에 TDD/PyQt/Cursor/UnitConverter 등 솔루션名 금지
- "~하면 좋겠다" 미래 가정 금지
- 성공 기준은 인터뷰에 나온 날짜·분·원·입력값 인용
- UnitConverter_XX 도메인 (meter/feet/yard, 발주·환산) 반영
```

---

## 2. R-G-I-O 정의 (참고)

| 항목 | 의미 | 작성 팁 |
|------|------|---------|
| **R** | Reality | 페르소나·업무·단위 출처 (카탈로그/발주서/메모) |
| **G** | Goal | "견적·발주 전 ~ 판정" — 행동 목표 |
| **I** | Issue | Mom Test 진짜 문제를 장애·증거 표로 풀기 |
| **O** | Outcome | 사용자가 원하는 **상태** (기능 목록 아님) |

---

## 3. 성공 기준 작성 템플릿

```
| # | 기준 | 측정 방법 | 기준선 (Mom Test) |
|---|------|-----------|-------------------|
| 1 | [오류/비용] | [기간·건수] | [날짜, 금액, 인용] |
| 2 | [시간] | [건별 처리 시간] | [분, 사건] |
| 3 | [입력/형식] | [Invalid 0회 등] | [실패 입력 예] |
| 4 | [연속 처리] | [재시작 횟수] | [재실행 사례] |
| 5 | [출력] | [m/mm 자릿수] | [후처리 분] |
```

## 4. STEP 1 연계 프롬프트 (선행)

```
Mom Test 인터뷰 종료 후:

다음만 출력:
1. 표면 문제 (잘못된 정의)
2. 진짜 문제 (한 문장)
3. Mom Test 증거 3줄

→ 완료 후 STEP 2 R-G-I-O 프롬프트 실행
```
